# PanGrab Pro 换服务器（迁移）指南

> 场景：后端已在旧服务器运行，需要迁移到新服务器（换 VPS/云主机、换机房等）。
> 目标：**用户账号、会员状态、订单、兑换码一条都不丢，且尽量不掉登录、不停服。**
>
> 后端 = Node + SQLite（默认 8787）。全部"状态"只在一个文件里：`server/data.db`。

---

## 一、迁移前必读（三个最关键的点）

### 1. 数据库文件 `data.db` 必须原样搬过去
所有用户、密码、会员到期时间、订单、兑换码都存在 `server/data.db`（SQLite 单文件）。
**不搬 = 所有人账号和会员全没了。** 这是迁移的核心。

### 2. `JWT_SECRET` 必须与旧服务器保持一致
用户登录后拿到的 token 是用 `JWT_SECRET` 签发的。
- **保持一致** → 用户无感迁移，不用重新登录 ✅
- **改了** → 所有已登录用户 token 失效，全部被登出（需重新登录，会员数据还在，但体验差）

所以：**把旧的 `.env` 整个复制到新服务器**，不要重新生成 `JWT_SECRET`。

### 3. 域名尽量不变（决定要不要重发扩展）
扩展里写死了后端地址（`extension/src/pro-config.js` 的 `apiBase`）。
- **沿用同一域名**（如 `api.你的域名`，只把解析指向新服务器 IP）→ 扩展**完全不用动** ✅ 强烈推荐
- **换新域名** → 必须改扩展 `apiBase` 并**重新打包/上架**，所有用户都要更新扩展才能用 ❌ 尽量避免

---

## 二、迁移方案总览

推荐**「保持同域名 + 数据整体搬迁」**，步骤：

```
旧服务器                          新服务器
  │ 1. 备份 data.db + .env           │
  │ ───────────── 传输 ───────────►  │ 2. 装环境、拉代码、npm install
  │                                  │ 3. 放入 data.db 和 .env
  │                                  │ 4. PM2 启动 + 本地 health 验证
  │                                  │ 5. 配反代 + HTTPS
  │ 6. DNS 把域名切到新服务器 IP ────►│（同域名，扩展不用改）
  │ 7. 更新支付回调地址（如域名变）    │
  │ 8. 验证无误后，下线旧服务器        │
```

---

## 三、详细步骤

### 步骤 0：迁移窗口建议
选低峰期操作。整个过程用户端影响很小（DNS 切换有几分钟缓存），云同步失败会自动重试。

### 步骤 1：在旧服务器备份数据与配置
```bash
cd ~/pangrab-pro/server
# 停一下写入更安全（可选）：pm2 stop pgpro
# 用 SQLite 官方在线备份，避免拷到写一半的文件：
sqlite3 data.db ".backup 'data-migrate.db'"      # 若没有 sqlite3，直接 cp data.db data-migrate.db 也可
cp .env env-backup

# 打包待传输
tar czf pgpro-migrate.tar.gz data-migrate.db env-backup
```
> 如果 stop 过服务，备份完可先 `pm2 start pgpro` 让旧服务继续跑，直到新服务器验证通过再下线。

### 步骤 2：把备份传到新服务器
```bash
# 在旧服务器执行（scp 到新服务器）
scp pgpro-migrate.tar.gz 用户名@新服务器IP:~/
```

### 步骤 3：新服务器准备环境 + 代码
```bash
# 装 Node 18+（建议用 nvm）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
# 重开终端后：
nvm install 20

# 拉代码
git clone https://github.com/HveinBlg/pangrab-pro.git
cd pangrab-pro/server
npm install
```
> `better-sqlite3` 是原生模块，需要能编译（新机若报错，装 `build-essential`/`python3`：
> Debian/Ubuntu: `sudo apt-get install -y build-essential python3`）。

### 步骤 4：放回数据与配置
```bash
cd ~
tar xzf pgpro-migrate.tar.gz
# 放到新代码的 server 目录，注意文件名对齐（DB_PATH 默认 ./data.db）
cp data-migrate.db ~/pangrab-pro/server/data.db
cp env-backup      ~/pangrab-pro/server/.env
```
> 检查 `.env` 里的 `PUBLIC_BASE_URL`：如果**域名不变**就无需改；如果换域名，改成新域名（见步骤 7）。

### 步骤 5：用 PM2 启动并本地验证
```bash
npm install -g pm2
cd ~/pangrab-pro/server
pm2 start src/index.js --name pgpro          # .env 会被 dotenv 自动加载
pm2 save
pm2 startup            # 按提示执行它输出的命令，实现开机自启
# 验证：
curl http://127.0.0.1:8787/api/health        # 期望 {"ok":true,...}
```

### 步骤 6：配置反向代理 + HTTPS（新服务器）
不要直接对外暴露 8787，用反代加 HTTPS。任选其一：

**Caddy（自动证书，最省心）** `/etc/caddy/Caddyfile`：
```
api.你的域名 {
    reverse_proxy localhost:8787
}
```
`sudo systemctl reload caddy`

**Nginx**（片段）：
```
server {
    server_name api.你的域名;
    location / { proxy_pass http://127.0.0.1:8787; proxy_set_header Host $host; }
    # 证书用 certbot 自动申请
}
```

**Lucky**：Web服务 → 添加规则 → 域名 `api.你的域名` → 反代目标 `http://127.0.0.1:8787` → 监听 443。

> 防火墙：放行 **443** 入站；**不要**对外开 8787（仅 127.0.0.1）。

### 步骤 7：切换域名解析 / 更新支付回调
**A. DNS（同域名迁移的关键一步）**
把 `api.你的域名` 的 A 记录从旧 IP 改成**新服务器 IP**。等生效（几分钟到几十分钟，取决于 TTL；可提前把 TTL 调小）。

**B. 支付回调地址（仅当 `PUBLIC_BASE_URL`/域名变化时才需要）**
- **支付宝**：开放平台里应用的**异步通知/回调地址**如指向具体域名，更新为新域名
- **Lemon Squeezy**：后台 Settings → Webhooks 的 Callback URL 更新为 `https://新域名/api/lemonsqueezy/webhook`
- 改完记得同步 `.env` 的 `PUBLIC_BASE_URL` 并重启：`pm2 restart pgpro`

> 域名不变则这步基本可跳过（回调地址天然仍指向同域名）。

### 步骤 8：验证 → 切换 → 下线旧机
```bash
# 从外网验证新服务器（走 HTTPS 域名）
curl https://api.你的域名/api/health         # {"ok":true}
```
逐项确认：
- [ ] `/api/health` 正常
- [ ] 扩展里能登录、能看到原有会员状态与收藏（说明 data.db 生效）
- [ ] 云同步上传/下载正常
- [ ] 发一个测试兑换码能激活（`/api/admin/codes`）
- [ ] 支付下单/回调正常（如已配支付）

**全部 OK 后**，再关停旧服务器（`pm2 delete pgpro` / 关机）。保留旧 `data.db` 备份至少几天以防万一。

## 四、扩展端要不要改？

| 情况 | 扩展是否需改 | 操作 |
|------|:---:|------|
| **域名不变**（只换 IP，DNS 切过去）| ❌ 不用改 | 无需任何操作，用户无感 |
| **换新域名** | ✅ 必须改 | 改 `extension/src/pro-config.js` 的 `apiBase` → 重新打包 → 上架/让用户更新 |

`pro-config.js`：
```js
self.PanGrabProConfig = { apiBase: "https://api.你的域名" };
```

> 结论：**能不换域名就不换域名**，迁移成本天壤之别。

## 六、常见问题

**Q：迁移后用户都被登出了？**
A：多半是 `JWT_SECRET` 变了。务必沿用旧 `.env` 的 `JWT_SECRET`。（会员数据不会丢，重新登录即可，但应避免。）

**Q：登录后看不到原来的收藏/会员？**
A：`data.db` 没搬成功，或 `DB_PATH` 指错了。确认新机 `server/data.db` 是旧库、且 `.env` 的 `DB_PATH` 与之匹配（默认 `./data.db`）。

**Q：`npm install` 报 better-sqlite3 编译错误？**
A：装编译工具链：`sudo apt-get install -y build-essential python3`，Node 用 18/20 LTS。

**Q：支付回调收不到、付款后不开通会员？**
A：检查 `PUBLIC_BASE_URL` 是否为新公网域名、支付宝/LS 后台回调地址是否更新、443 是否可从公网访问、`pm2 restart` 是否执行。

**Q：想零停机迁移？**
A：先在新机跑起来并验证（此时 DNS 仍指旧机），最后只切 DNS。切换期间两边都能服务，云同步失败也会自动重试，影响极小。

## 七、日常运维（迁移后）
- **定时备份** `data.db`（强烈建议）：
  ```bash
  # crontab -e，每天凌晨 3 点备份并保留
  0 3 * * * cd ~/pangrab-pro/server && sqlite3 data.db ".backup '/root/backup/pgpro-$(date +\%F).db'"
  ```
- 常用 PM2：`pm2 logs pgpro`（看日志）、`pm2 restart pgpro`（重启）、`pm2 status`
- 更新代码：`git pull && npm install && pm2 restart pgpro`

## 八、甲骨文云（Oracle Cloud）专属步骤 ⭐

> 通用流程见第一~五节，本节只讲甲骨文特有、最容易踩的坑。**照这里做能少折腾几小时。**

### 8.1 选实例（Compute Instance）
- **机型**：
  - **ARM（Ampere A1）**：免费额度大（最高 4 OCPU / 24GB 内存），跑这个后端绰绰有余。**但架构是 aarch64**，`better-sqlite3` 需在 arm64 上编译（见 8.4）。免费 ARM 常提示 "Out of capacity"，多换可用域(AD)或换时段重试。
  - **x86（AMD, VM.Standard.E2.1.Micro）**：架构通用、无编译顾虑，但只有 1GB 内存，也够用。
- **系统镜像**：建议 **Ubuntu 22.04**（本指南命令以 Ubuntu 为准）。
- **SSH 密钥**：创建时下载好私钥，用于登录。

### 8.2 ⚠️ 双层防火墙（甲骨文头号大坑）
甲骨文的端口拦截是**两层**，只开一层 443 依然不通：

**第 1 层：云控制台（VCN 安全列表 / NSG）**
- 进入实例 → 子网(Subnet) → 默认安全列表(Security List) → **添加入站规则(Ingress Rules)**：
  - 源 `0.0.0.0/0`，协议 TCP，目标端口 **443**
  - 再加一条目标端口 **80**（Caddy/Certbot 申请证书要用）

**第 2 层：实例系统内的 iptables（最容易被忽略）**
甲骨文的 Ubuntu 镜像自带 iptables 规则，**默认只放行 SSH(22)，其它全 DROP**。必须手动放行：
```bash
sudo iptables -I INPUT 6 -p tcp --dport 443 -j ACCEPT
sudo iptables -I INPUT 6 -p tcp --dport 80  -j ACCEPT
sudo netfilter-persistent save        # 持久化，重启不丢
# 若没有该命令：sudo apt-get install -y iptables-persistent
```
> **两层都放行后 443 才通。** 排查"443 连不上"时，先确认这两层。

### 8.3 固定公网 IP（Reserved Public IP）
默认分配的是**临时公网 IP**，实例停止/重启后可能变。
- 实例 → 附加的 VNIC → IPv4 地址 → 把公网 IP 从 **Ephemeral 改为 Reserved（保留）**。
- 否则 IP 一变，DNS 又得重配。

### 8.4 装环境（ARM 注意编译）
```bash
sudo apt-get update
sudo apt-get install -y build-essential python3 git   # better-sqlite3 原生编译依赖
# 装 Node 20 LTS（arm64/x86 都有官方包）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
# 重开终端后：
nvm install 20
```
后续拉代码、放 `data.db`/`.env`、PM2 启动，完全照第三~五节执行。

### 8.5 反代 + HTTPS（甲骨文推荐 Caddy）
甲骨文机器公网直连、Caddy 自动申请 Let's Encrypt 证书最省心：
```bash
# 安装 Caddy（Ubuntu）
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update && sudo apt install -y caddy
```
`/etc/caddy/Caddyfile`：
```
api.你的域名 {
    reverse_proxy localhost:8787
}
```
`sudo systemctl reload caddy` 即可（80/443 两层防火墙都放行后，证书才能自动签发成功）。

### 8.6 DNS 切换（保持同域名 = 扩展不用改）
- 你用的是 **Cloudflare**：进 DNS → 把 `api.你的域名` 的 **A 记录改成甲骨文的保留公网 IP**。
- **Cloudflare 代理开关**：Caddy 自动申请证书期间，建议先把该记录设为 **灰云(DNS only)**，证书签发成功后再按需开橙云。
- 只要**域名不变**，`pro-config.js` 和扩展**完全不用动**，用户无感。

### 8.7 甲骨文专属 FAQ
**Q：443 端口在浏览器/curl 就是连不上？**
A：99% 是漏了 8.2 的**第 2 层 iptables**。云安全列表开了不够，系统内还得放行。

**Q：免费 ARM 一直 "Out of host capacity" 创建不了？**
A：甲骨文免费 ARM 容量紧张。换实例所在**可用域(AD)**、换个时段、或用脚本轮询重试；实在不行先用 x86 micro 顶着。

**Q：Caddy 申请证书失败？**
A：确认 80 端口两层都放行、域名 A 记录已指向本机 IP 且 Cloudflare 为灰云(DNS only)、`PUBLIC_BASE_URL` 与域名一致。
